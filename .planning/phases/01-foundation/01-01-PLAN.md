## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Django login view | Untrusted email+password credentials cross here |
| Django session → authenticated views | Session cookie is the trust token — must be HttpOnly and Secure in prod |
| Django login → PostgreSQL | User lookup with email; SQL injection surface |
| HTMX request → Django CSRF middleware | All HTMX POST/PUT/DELETE requests must include X-CSRFToken header |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | `/accounts/login/` view | mitigate | Django `authenticate()` uses constant-time password comparison (PBKDF2). No timing oracle. |
| T-01-02 | Tampering | HTMX POST requests | mitigate | `CsrfViewMiddleware` + `htmx:configRequest` event injects `X-CSRFToken` header from `<meta name="csrf-token">` on every HTMX request. Verified in acceptance criteria. |
| T-01-03 | Elevation of Privilege | Inactive user login | mitigate | Django `authenticate()` checks `is_active` before returning user. Explicit error message "Esta conta está inativa" shown to user. |
| T-01-04 | Elevation of Privilege | Open redirect on `next=` parameter | mitigate | Django validates `next` parameter against safe URL list via `is_safe_url()`. Never pass untrusted `next` values through template. |
| T-01-05 | Information Disclosure | Login error messages | accept | Error message does not distinguish "email not found" from "wrong password" — both show the same string. No user enumeration. |
| T-01-06 | Elevation of Privilege | Session fixation | mitigate | Django calls `cycle_key()` automatically after successful login. No additional code needed. |
| T-01-07 | Elevation of Privilege | SECRET_KEY exposure | mitigate | SECRET_KEY read from `.env` via `python-decouple`. Never hardcoded. `.env` not committed (in .gitignore). |
| T-01-08 | Denial of Service | Unlimited login attempts | accept | No rate limiting in Phase 1 — 20 internal users, no public surface. Django admin brute force protection is low risk at this scale. Add rate limiting in Phase 2 if threat model changes. |
| T-01-SC | Tampering | pip package installs | mitigate | All 8 packages passed slopcheck [OK]. No [ASSUMED] or [SUS] packages. Package Legitimacy Audit in 01-RESEARCH.md confirms clean bill. |
