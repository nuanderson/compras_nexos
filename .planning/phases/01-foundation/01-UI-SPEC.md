# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Foundation (authentication, user management, organizational units).
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
> This is the foundation design system — all tokens and patterns declared here apply to all subsequent phases.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — custom CSS only |
| Icon library | none — Unicode/HTML entities or inline SVG only in Phase 1 |
| Font | system-ui, -apple-system, "Segoe UI", Roboto, sans-serif |

**Rationale:** Stack is Django 5.2 + HTMX 2.0 with server-rendered templates. No Tailwind, no Bootstrap, no JavaScript framework. Styling delivered via a single `static/css/main.css`. No third-party component registry — no vetting gate required.

| sm | 8px | Compact form element internal padding, inline element gaps |

| lg | 24px | Section padding inside cards, form group vertical gap |
| xl | 32px | Card-to-card gap, major layout gaps |

**Exceptions:**
- Sidebar width: 240px (fixed)

## Component Contracts

### 1. Login Page

**Layout:** Full viewport, `display: flex; align-items: center; justify-content: center; min-height: 100vh; background: var(--color-bg)`.

**Card:** `max-width: 400px; width: 100%; padding: 32px; background: var(--color-surface); border: 1px solid var(--color-border); border-radius: 8px`.

**Elements (top to bottom):**
1. Logo/brand: "ComprasNexos" — 20px/semibold, centered, margin-bottom 32px
2. Page-level heading: "Acesse sua conta" — 14px/400/muted, centered, margin-bottom 24px
3. Email field: label "E-mail" + `<input type="email">`, full width
4. Password field: label "Senha" + `<input type="password">`, full width, margin-top 16px
5. Forgot password link: "Esqueceu a senha?" — 12px, right-aligned, color: accent, margin-top 8px
6. Submit button: "Entrar na conta" — full width, accent fill, 44px height, margin-top 24px
7. Error banner (conditional): red border card above form fields when credentials fail

**Input style:** `width: 100%; height: 44px; padding: 0 16px; background: var(--color-bg); border: 1px solid var(--color-border); border-radius: 6px; color: var(--color-text); font-size: 14px`. Focus: `outline: 2px solid var(--color-accent); outline-offset: 2px`.

**HTMX behavior:** None on login page — standard Django form POST. Password reset link navigates to `/accounts/password-reset/`.

---

### 2. Password Reset Flow

**Two pages:**

**Page 1 — Request reset:** Same card layout as login. Fields: email only. CTA: "Enviar link de redefinição". After submit: replace form with success message (Django messages framework — no HTMX needed).

**Page 2 — Set new password:** Same card layout. Fields: new password + confirm password. CTA: "Redefinir senha". After success: redirect to login with Django success message.

**Validation errors:** Rendered server-side by Django's form error display. Error text in `var(--color-destructive)`, 12px, below the field.

### 8. Forms (Admin panel — user/unit CRUD)

**Form container:** Card layout (surface bg, border, 8px radius), `max-width: 640px`, `padding: 32px`.

**Field group:** `display: flex; flex-direction: column; gap: 8px; margin-bottom: 16px`.

**Label:** `font-size: 12px; font-weight: 600; color: var(--color-text-muted); text-transform: uppercase; letter-spacing: 0.05em`.

**Input/Select/Textarea:** Same as login input style. Select uses system appearance.

## Interaction Contracts

### HTMX Patterns Used in Phase 1

| Interaction | Pattern | Target | Swap |
|-------------|---------|--------|------|
| Admin form submit (create/edit user) | `hx-post` | `#form-container` | `innerHTML` |
| Admin form submit (create/edit unit) | `hx-post` | `#form-container` | `innerHTML` |
| Deactivate user — load confirmation | `hx-get` | `#confirm-container` | `innerHTML` |
| Deactivate user — execute | `hx-post` | `#user-row-{id}` | `outerHTML` |
| Deactivate unit — load confirmation | `hx-get` | `#confirm-container` | `innerHTML` |
| Cancel confirmation | `hx-get` (empty partial) | `#confirm-container` | `innerHTML` |

**CSRF:** `<meta name="csrf-token" content="{{ csrf_token }}">` in `<head>`. HTMX config in `base.html` body sends `X-CSRFToken` header on all HTMX requests via `htmx:configRequest` event listener.

**No polling in Phase 1** — no real-time status updates needed. Polling introduced in Phase 2 for requisition status.

**`hx-boost`:** Enabled on `<nav class="sidebar">` to turn nav links into partial-page AJAX loads. Content area replaces without re-loading sidebar or top bar.

### Focus Management

After HTMX swap that replaces a form with a success state: no automatic focus needed. After an error swap that re-renders the form: focus remains on the last field (browser default behavior).

### Loading States

HTMX adds `.htmx-request` class to the element that triggered the request. All primary buttons get:
```css
.htmx-request { opacity: 0.6; cursor: not-allowed; pointer-events: none; }
```
No spinner required in Phase 1 — server response is fast for 20-user internal tool.

## Accessibility Contract

- All form inputs have explicit `<label for="...">` associations — no `placeholder` as substitute for label
- Color is never the sole differentiator — status badges include text ("Ativo", "Inativo")
- All interactive elements keyboard-reachable via Tab order
- Focus ring: `outline: 2px solid var(--color-accent); outline-offset: 2px` — never `outline: none`
- Nav landmark: `<nav>` for sidebar, `<main>` for content, `<header>` for top bar
- Error messages: associated with field via `aria-describedby`
- Buttons that only contain icons (if any): `aria-label` required

## CSS Architecture

**Single file:** `static/css/main.css`

**Structure:**
1. `:root` — all CSS custom properties (tokens)
2. CSS reset (minimal — box-sizing, margin 0, line-height)
3. Layout primitives (`.layout`, `.sidebar`, `.main`, `.topbar`, `.content`)
4. Typography utilities
5. Components (`.card`, `.table`, `.btn`, `.badge`, `.form-group`, `.input`, `.banner`)
6. State modifiers (`.htmx-request`, `.is-active`)

**No utility classes beyond what is listed.** No Tailwind, no Bootstrap. No CSS-in-JS.
